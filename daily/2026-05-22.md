## AI 基础 — 智能体（Agent）（70 分钟）

### 本章核心一句话

> Agent 的关键不在「像人」，而在于能不能在**明确权限和可审计流程**里，把建议推进到行动。

### 学习重点

**1. 第一性原理：Agent 是被约束的执行循环（10 分钟）**
- LLM 一次回答问题，只是生成文本；Agent 更进一步：拆任务、查资料、调用 API、生成操作草稿、等待反馈再继续
- 错误也从「答错」升级为「做错」——一旦能调用工具、写入系统、触发支付，模型误判就有真实代价
- 完整的 Agent 必须具备：**目标、工具、状态、权限、停止条件**，缺一不可
- 分工原则：模型负责提出候选行动，系统负责限制行动空间，用户负责批准高风险边界

**2. 五个知识节点（20 分钟）**

| 节点 | 一句话理解 |
|------|-----------|
| Tool Use | Agent 调用外部能力（搜索、数据库、API、支付）；工具让 Agent 从「会回答」变成「能做事」，也让 Prompt Injection 的后果从文本错误变成真实操作 |
| Planning | 把目标拆成步骤；计划是候选路线，不是授权——越靠近高风险动作，越需要系统规则拆开检查 |
| State | 任务进度、工具结果、用户确认记录；必须外置可查，不能只藏在模型上下文里 |
| Reflection | Agent 检查自己的中间结果并修正；可以提升质量，但不能替代外部验证——模型可能给自己的错误找理由 |
| Multi-Agent | 多个 Agent 分工协作；适合复杂工作流，但会放大协调问题：上下文传递丢失、权限扩散、错误被当成事实 |

**3. Tool Use 设计要点（20 分钟）**

设计一个工具时必须明确：
- 输入 schema 是什么
- 权限范围有多大
- 是否只读，是否产生外部副作用
- 调用前后如何记录
- 哪些调用需要人工确认

读数据、写数据库、发送请求、改配置——**不是同一风险等级**。

**4. 在 AI × Web3 里的位置（20 分钟）**

一个相对稳的 AI × Web3 Agent 架构：

```
① 用户提出目标和约束
② Agent 读取上下文并生成计划
③ 系统把计划拆成只读步骤和候选写入步骤
④ 只读工具自动执行，写入工具进入 policy 检查
⑤ Simulation 展示链上影响
⑥ 用户确认高风险动作
⑦ Wallet / Smart Account 执行
⑧ 日志记录每一步和最终状态
```

最危险的设计：Agent 同时拥有**模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限**。

### 我的笔记（学完后填写）

**Handbook 说「工具让 Agent 从会回答变成能做事，但也让错误从答错变成做错」——你怎么理解这个升级？**
需要更谨慎对待开发环境，以及授予的权力边界和清楚的需求描述，因为一旦某一环节出错，所做或者所改的地方就可能天差地别，虽然能进行版本回退，但是同样付出了时间和token的成本

> **补充（Claude）：** 在 Web3 场景里「版本回退」这个选项本身就不存在——合约执行和链上交易一旦上链就不可撤回。所以代价比普通软件还要重：不是「回退有成本」，而是「根本没有回退」。这也是为什么 Handbook 把停止条件和人工确认放得那么重——Agent 在链上的权力越大，边界设计就越是前提条件，而不是可选项。

**8 步架构里，你认为哪一步是最关键的安全节点？为什么？**
3，需要很精准的区分这两部分，否则有可能出现prompt injection的破坏

> **补充（Claude）：** ③ 的理由成立。另一个同等重要的节点是 ⑥（用户确认）——③ 是系统层的分类拦截，防止写入工具被误触发；⑥ 是人类层的最终判断，防止系统层分类出错后仍有最后一道关卡。两者一起构成「纵深防御」：任何一层单独失守，另一层还在。

## Web3 基础 — 网络（Network）（20 分钟）

### 本章核心一句话

> 不同链有不同的安全假设和性能取舍——选网络不是选「哪个更好」，而是选「哪个的权衡适合我的场景」。

### 学习重点（轻量阅读，建立直觉）

**主要网络类型：**

| 类型 | 代表 | 一句话理解 |
|------|------|-----------|
| Layer 1 | Ethereum、Solana | 基础结算层，安全性最高，速度和成本是瓶颈 |
| Layer 2 | Arbitrum、Optimism、Base | 继承 L1 安全性，通过批量处理提升速度、降低 Gas |
| Rollup | Optimistic / ZK Rollup | L2 的两种主流实现：乐观型假设无欺诈，ZK 型用数学证明正确性 |
| 测试网（Testnet） | Sepolia、Base Sepolia | 和主网行为一致，代币无价值，用于开发验证 |

**选网络的三个维度：**

| 维度 | 问题 |
|------|------|
| 安全性 | 这笔资产/操作需要多高的最终确定性？ |
| 成本 | Gas 费是否在用户可接受范围内？ |
| 生态 | 需要用到的协议/合约部署在哪条链上？ |

### 我的笔记（学完后填写）

**用自己的话解释：Layer 2 怎么在「继承 L1 安全性」的同时降低费用？**
通过把证明和结果保留主网，其他的执行跑L2的方式可以保留安全的同时极大的提升了交易确认速度（确认速度更快），gas费也降低，但是牺牲了一小部分的安全性和多出来的复杂性（没有什么是两全其美的,q,q）

> **补充（Claude）：** L2 的设计目标是**完整继承** L1 安全性，不是降低它——所以「牺牲了一小部分安全性」这个说法需要修正。真正的取舍是：Optimistic Rollup 有 7 天提款等待期（挑战窗口期），ZK Rollup 需要大量计算来生成数学证明。代价是**延迟和复杂性**，不是安全性本身。
