# AI 基础概念整理

> 作者：Ezekiel | 日期：2026-05-27
> 用自己的话整理 AI 基础概念，辅助建立后续学习 Agent、workflow、AI coding 和工具调用的共同语言。
> 使用 AI 辅助纠错和补充，复核说明见文末。

---

## 1. LLM（大语言模型）

**一句话解释：**
LLM 是把文本切成 token 再转成向量，根据概率分布逐个生成下一个 token 的语言模型——它给出的是「统计上最可能」的续写，不是「最正确」的答案。

**具体例子：**
你问「以太坊的创始人是谁」，LLM 不是在数据库里查找，而是根据训练数据里这类句子的统计规律，生成「Vitalik Buterin」——如果训练数据里有错误，生成的答案也可能是错的。

**常见误区：**
以为 LLM 在「理解」问题或「知道」答案。实际上它只是在做高维度的概率预测，生成的每个 token 都是当前上下文下最高概率的选择，不保证事实正确。

---

## 2. Prompt（提示词）

**一句话解释：**
Prompt 是给模型的完整输入——不只是问题本身，还包括角色定义、任务规则、输出格式、禁止行为和示例，是软约束而非安全边界。

**具体例子：**
「帮我翻译这句话」是一个很弱的 prompt；「你是一名专业翻译，请把以下技术文档翻译成中文，保留所有代码块不翻译，如果有歧义请注明原文」是一个结构化的 prompt，能大幅提升输出质量和稳定性。

**常见误区：**
以为 prompt 是安全边界。「你不能做 X」写在 prompt 里只是软约束，攻击者可以通过 Prompt Injection 绕过，真正的安全必须在代码层和权限层实现。

---

## 3. Context Window（上下文窗口）

**一句话解释：**
Context window 是模型生成每一个 token 时能「看到」的全部内容范围——系统 prompt、对话历史、工具返回、检索文档都在里面，窗口之外的内容对模型来说不存在。

**具体例子：**
Claude 的 context window 约 200K tokens。如果一段对话历史超出这个范围，早期的内容就会被截断，模型无法参考——这就是为什么长对话需要摘要压缩或 RAG 策略。

**常见误区：**
把 context window 理解为模型的「记忆」。模型没有持久记忆，窗口里的内容只在当前这次推理中有效，下一次对话从头开始，必须重新注入需要的上下文。

---

## 4. Workflow（工作流）

**一句话解释：**
Workflow 是把复杂目标拆分成有序步骤的执行结构——每个步骤可以由模型、工具或人来完成，整体流程有明确的状态、分支和停止条件。

**具体例子：**
「分析一个 DeFi 协议的风险」可以被拆成：① 读取协议文档 → ② 检索历史漏洞 → ③ 生成风险清单 → ④ 人工确认 → ⑤ 输出报告。每一步有明确的输入输出，失败时知道从哪里重试。

**常见误区：**
以为 workflow 就是让 Agent 自由发挥。好的 workflow 恰恰相反——它需要显式定义每个步骤的工具、权限和确认节点，越关键的步骤越不能让 AI 自主决策。

---

## 5. Agent（智能体）

**一句话解释：**
Agent 是围绕目标持续调用工具、读取状态、调整步骤的 AI 执行系统——它的关键不是「更聪明」，而是在明确权限和可审计流程里把建议推进到行动。

**具体例子：**
一个 DeFi 监控 Agent：持续读取链上仓位数据 → 检测到抵押率接近清算线 → 生成再平衡交易草案 → 等待用户确认 → 执行。每一步都有工具调用和状态记录，不是一次性回答。

**常见误区：**
以为 Agent 的核心是「多模态能力」或「更多功能」。Agent 的关键其实是执行循环的边界设计：目标、工具、状态、权限、停止条件缺一不可——没有边界的 Agent 是最危险的 Agent。

---

## 6. Tool Use（工具调用）

**一句话解释：**
Tool Use 是模型调用外部函数或接口的能力——搜索、数据库查询、代码执行、API 调用、链上交易，让模型从「会回答」变成「能做事」，也让模型误判的后果从「答错」变成「做错」。

**具体例子：**
Agent 被要求「查一下 ETH 当前价格」：模型生成工具调用指令 `get_price("ETH")`，工具执行后返回结果，模型再把结果整合进回答。这个过程模型自己不知道价格，是工具提供的。

**常见误区：**
把 MCP 和 Skill 都归为「工具」。MCP 是工具调用的协议层（定义如何调用），Skill 是模型的指令层（告诉模型什么时候用哪个工具、怎么规划）——两者工作在不同层级，不是同一件事。

---

## 7. AI Coding（AI 辅助编程）

**一句话解释：**
AI Coding 是人和 AI Coding Agent 共同迭代软件的工作方式——人负责目标、约束和验收，Agent 负责搜索代码、生成 patch、跑测试、解释错误，把工程反馈循环压短。

**具体例子：**
用 Claude Code 修一个 bug：说明报错信息和期望行为 → Agent 搜索相关文件 → 生成最小 patch → 跑测试确认 → 查看 git diff 审查。整个过程 Agent 参与执行，人负责最终判断。

**常见误区：**
以为 AI Coding 就是「把需求丢给 AI 等代码出现」。真正可用的做法是任务要小、上下文要准、验证要硬——速度变快之后真正的挑战是能不能描述清楚任务、审查结果和控制改动范围。

---

## 8. Guardrails（护栏）

**一句话解释：**
Guardrails 是在 AI 系统执行链路上设置的约束层——限制模型能输出什么、Agent 能调用哪些工具、哪些动作需要人工确认，把 AI 的行动空间限制在可接受范围内。

**具体例子：**
一个 DeFi Agent 的 guardrail：每次 swap 金额不超过 $100、只能调用白名单内的合约、滑点超过 2% 自动拒绝、任何写入操作先 simulation 再 human check。这些规则不在 prompt 里，而是在系统层强制执行。

**常见误区：**
以为 prompt 里写了「不能做 X」就是 guardrail。真正的 guardrail 是代码层和权限层的硬约束，不依赖模型是否「理解」规则——模型可以被 Prompt Injection 绕过，系统层的检查不会。

---

## 9. Tracing（追踪）

**一句话解释：**
Tracing 是记录 AI 系统每一步执行过程的可观测能力——哪个模型、调用了哪个工具、传入什么参数、返回什么结果、耗时多少、是否失败，让系统行为可被审计和调试。

**具体例子：**
一个 Agent 执行了五步工具调用，最后输出了错误结果。有 tracing 就能看到是第三步的 RAG 检索拿到了错误文档，进而修复；没有 tracing 只能重新运行猜问题在哪。

**常见误区：**
把 tracing 和区块链「交易（transaction）」混淆——两个完全不同的概念。Tracing 是软件可观测性领域的术语，指的是程序执行路径的记录，和链上交易没有关系。

---

## 10. Human-in-the-Loop（人工确认节点）

**一句话解释：**
Human-in-the-loop 是在 AI 执行链路的关键节点暂停，等待人类审查和确认后再继续——核心是把高风险动作的最终决策权留给人，而不是让 Agent 全自动执行。

**具体例子：**
AI Agent 生成了一笔链上转账交易草案，simulation 显示会转出 5 ETH。此时系统暂停，把交易详情展示给用户：目标地址、金额、预计 gas——用户确认后才签名执行，拒绝则取消。

**常见误区：**
以为 human-in-the-loop 就是「让人随时干预」。实际上它是精确设计的——哪些步骤自动执行，哪些步骤必须人工确认，取决于风险级别和业务规则，不是所有步骤都需要人参与，过度确认反而降低可用性。

## AI 辅助复核说明

以下是我使用 AI 辅助整理时，AI 指出并纠正的主要问题：

| 概念 | 我的原始理解 | 问题 | 纠正方向 |
|------|------------|------|---------|
| LLM | 「根据可能的权重给出最符合的回答」 | 「最符合」容易误解为「最正确」，实际是「概率最高」，不保证正确 | 改为「统计上最可能的续写」，强调概率而非正确性 |
| Prompt | 「输入的需求/问题部分」 | Prompt 不只是问题，还包括系统规则、角色、格式要求、示例 | 扩展为「完整输入」，包含所有给模型的指令 |
| Context Window | 「AI 记住的最长记忆内容」 | 「记住」会误导为持久记忆，实际是单次推理的可见范围 | 改为「能看到的全部内容范围」，强调无持久性 |
| Agent | 「多模态能力，可以编码」 | 多模态不是 Agent 的定义特征；Agent 的核心是执行循环和边界设计 | 聚焦「围绕目标持续调用工具、在明确权限里推进行动」 |
| Tool Use | 「MCP、Skills 等等」 | MCP 是协议层，Skill 是指令层，两者不是工具本身 | 区分工具（外部函数调用）、协议（MCP）、指令层（Skill） |
| Workflow | 「不清楚」 | 补充完整 | 工作流 = 把复杂目标拆成有序步骤，每步有状态、分支和停止条件 |
| Guardrails | 「不清楚」 | 补充完整 | 系统层的硬约束，不是 prompt 里的软规则 |
| Tracing | 「AI 交易？」 | 完全理解错误，tracing 是软件可观测性术语，与区块链交易无关 | 改为「记录 AI 系统执行过程的可观测能力」 |
| Human-in-the-loop | 「不清楚」 | 补充完整 | 在高风险节点暂停等待人工确认，不是随时干预 |
