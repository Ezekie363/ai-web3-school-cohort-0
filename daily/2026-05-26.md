## AI 基础 — 评估（Evaluation）（70 分钟）

### 本章核心一句话

> Evaluation 是把「感觉效果不错」变成「**系统可持续改进**」的方法——没有 eval，任何改动都只能靠主观试用判断，迟早被回归问题拖住。

### 学习重点

**1. 第一性原理：不能被重复测量的 AI 行为，就不能被稳定改进（10 分钟）**
- AI 输出有概率性，没有固定样本和评估标准，无法判断变化来自改进、运气还是样本太少
- 三个核心原则：
  - **先测任务，不只测模型**：用户关心的是整条链路是否完成任务，不是榜单分数
  - **先保住关键失败场景**：高风险错误、常见问题、边界条件要先进 regression set
  - **评估要贴近产品**：离真实输入越远，eval 越容易变成自我安慰

**2. 五个知识节点（20 分钟）**

| 节点 | 职责 | 关键要素 | 注意点 |
|------|------|---------|-------|
| Harness | 运行 eval 的框架：喂样本、调用系统、收集输出、运行 grader、记录结果 | 输入样本 + 评分规则 + 系统版本 + 配置 + 日志 + 报告 | 价值在于**可重复**——没有可重复运行的 eval，就无法比较不同改动 |
| Golden Set | 被认真挑选和标注的测试样本集 | 30-100 条：常见问题 + 边界问题 + 容易误判 + 高风险 + 历史 bug + 真实反馈 | 质量 > 数量；每修一个重要 bug，都应考虑把它加进来 |
| LLM-as-Judge | 用模型给模型输出评分，适合开放式答案 | 规则评分（可自动判断字段）+ LLM judge（开放式质量）+ 人工抽检（高风险） | Judge 模型也会偏、会漏、会被输出风格影响；定期校准与人工评分的一致性 |
| Regression | 防止旧问题复发——每次改动都重跑历史失败样本 | 用户反馈错误 → 复现记录 → 标注期望输出 → 加入 set → 每次发布前跑一次 | AI 应用极容易「修 A 坏 B」，regression 是唯一可靠的防线 |
| Observability | 线上观察真实系统行为（eval 在发布前，observability 在使用中） | 输入来源 + 检索结果 + 工具调用 + 模型输出 + 错误重试 + 用户反馈 + 成本延迟 | 没有 observability，不知道真实用户在哪里失败，也不知道该往 golden set 里补什么 |

**3. Regression 的实用流程（20 分钟）**

```
用户反馈一个错误
→ 复现并记录输入
→ 标注期望输出或拒答条件
→ 加入 regression set
→ 之后每次发布前跑一次
```

**4. 在 AI × Web3 里的位置（20 分钟）**

需要特别评估的场景：

| 场景 | 评估重点 |
|------|---------|
| 交易解释 | 是否准确，是否遗漏关键风险 |
| 风险提示 | 是否漏报高风险动作 |
| 工具调用 | 参数是否越界，是否调用了不该调用的工具 |
| 拒绝能力 | 是否能拒绝不确定请求，是否能识别 Prompt Injection |
| 引用追溯 | 来源是否可追溯 |
| 高风险动作 | 是否正确触发 human check |

Eval 不替代交易模拟和权限控制，但能让你**持续发现系统在什么场景下不可靠**。

## Web3 基础 — 索引（Indexing）（20 分钟）

### 本章核心一句话

> 链上数据是公开的，但不等于好用——Indexing 把区块、交易、事件和合约状态整理成产品和 AI Agent 能**快速查询的结构化数据**。链上是事实来源，索引层是可用数据层。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- 产品关心的是「这个地址的仓位」「这个协议的 TVL」，而不是原始区块流——索引层负责把底层事实转换成这些查询对象
- **事件是重要入口**：合约 event 是索引器构建状态的主要信号
- **RPC 不是数据库**：适合读取链状态和发送交易，不适合承载复杂历史查询（有 rate limit、速度慢）
- **索引要能重放**：合约升级、reorg、bug 修复时，需要从某个区块重新构建数据

**四个核心概念：**

| 概念 | 一句话理解 | 关键问题 |
|------|-----------|---------|
| Event Indexing | 监听合约日志（Transfer/Deposit/VoteCast 等），整理成可查询记录 | event 是否包含关键地址？是否有 indexed 参数？失败交易不产生成功 event |
| Subgraph | 声明式定义如何索引合约事件，通过 GraphQL 暴露查询接口（The Graph） | 合约地址变更、事件结构变化、reorg、同步延迟都会影响数据质量 |
| RPC | 应用和节点交互的接口：读取链状态、查询日志、发送交易 | rate limit、节点不同步、archive 数据不可用、查询范围过大 |
| Data Pipeline | 把链上数据、链下数据、索引结果组合成可分析/可监控/可供 AI 使用的数据系统 | Agent 拿到过期索引或错误解码，后续推理再强也建立在错误事实上 |

**对 AI Agent 的意义：**
- 好的索引层给 Agent 提供：**结构化 + 带来源 + 带时间戳 + 可回溯**的数据
- 模型负责解释和推理，索引层负责提供事实——事实层出问题，推理层再好也没用

### 我的笔记（学完后填写）

**Handbook 说「RPC 不是数据库」——什么时候用 RPC，什么时候需要索引层？**
RPC只适合实时性的查询某一笔交易的信息，不适合大批量的超广的区块搜索，当需要数据分析（tvl、活跃地址，tx count）这类信息才需要索引层（因为rpc有limit rate）

> **补充（Claude）：** 理解正确。可以再加两个判断维度：① **聚合需求**——RPC 返回的是原始数据（一个 event、一个区块），不能做 GROUP BY、SUM、过滤；索引层可以提前聚合好，给产品直接用。② **并发和延迟**——用户量上来后，大量前端直接打 RPC 节点会被 rate limit 或响应变慢；索引层天然作为缓冲。简单记：**实时单点查询用 RPC；历史、聚合、高并发查询用索引层**。
