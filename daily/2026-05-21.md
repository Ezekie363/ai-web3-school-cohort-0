## AI 基础 — 检索增强生成（RAG）（70 分钟）

### 本章核心一句话

> RAG 不是让模型「记住更多」，而是在生成时**按需取用**可信数据——把知识管理和语言生成分开。

### 学习重点

**1. RAG 解决什么问题？（10 分钟）**
- LLM 的训练数据有截止日期，无法知道最新信息
- 把所有知识塞进 Prompt 超出 Context Window 且成本极高
- RAG 的思路：先检索相关片段，再把片段注入上下文，让模型基于检索结果生成答案

**2. 四个关键概念（20 分钟）**

| 概念 | 一句话理解 |
|------|-----------|
| Embedding | 把文本转换为数字向量，语义相近的文本在向量空间中距离近 |
| 向量数据库（Vector DB） | 专门存储和检索 Embedding 的数据库，支持「找最相似的 N 条」 |
| Chunk | 将长文档切割成小段落，是检索和注入上下文的基本单位 |
| 相关性分数（Relevance Score） | 检索时衡量文档块与查询语义相似度的分数，决定哪些内容被选入上下文 |

**3. RAG 的完整流程（20 分钟）**

```
离线阶段：文档 → 切块（Chunk）→ Embedding → 存入向量数据库

在线阶段：用户提问 → Embedding → 向量检索 → 取回 Top-K 相关块
         → 注入上下文 → 模型生成答案
```

关键决策点：
- Chunk 太大：上下文噪音多，相关性被稀释
- Chunk 太小：缺乏完整语义，答案片面
- Top-K 太多：超出 Context Window 或引入无关内容

**4. 在 AI × Web3 里的位置（20 分钟）**
- RAG 可以检索链上数据：合约 ABI、历史交易记录、协议文档
- 攻击面：**RAG 投毒（RAG Poisoning）**——向量数据库里植入恶意文档块，被检索后污染上下文
- 防御：对检索来源做信任评级，关键操作的数据来源必须是可验证的链上事实，而非 RAG 结果

### 我的笔记（学完后填写）

**用自己的话解释：RAG 和「把所有文档塞进 Prompt」有什么本质区别？**
最关键的一点是存入数据库进行规范化的切片整理

**RAG Poisoning 可能在什么场景下发生？你能举一个具体例子吗？**
信息来源的不准确，一些未经证实的信息面被植入从而污染上下文

RAG本质上是agent的知识库，语料库？

## Web3 基础 — 开发栈（Dev Stack）（20 分钟）

### 本章核心一句话

> Web3 开发栈是一套专为「和链交互」设计的工具链——从本地模拟链到主网部署，每一层都有对应工具。

### 学习重点（轻量阅读，建立直觉）

**四层工具栈：**

| 层级 | 工具 | 作用 |
|------|------|------|
| 合约语言 | Solidity / Vyper | 编写智能合约的编程语言 |
| 开发框架 | Hardhat / Foundry | 编译、测试、部署合约；Foundry 用 Solidity 写测试，速度更快 |
| 本地链 | Anvil / Hardhat Network | 在本地模拟链上环境，无需真实 Gas，方便开发调试 |
| 链交互库 | ethers.js / viem | 前端或脚本与链上合约通信的 JavaScript 库 |

**开发工作流（从写代码到上链）：**

```
写合约（Solidity）→ 本地测试（Foundry/Hardhat）
→ 测试网部署（Sepolia 等）→ 审计 → 主网部署
```

**两个重要概念：**

| 概念 | 一句话理解 |
|------|-----------|
| 测试网（Testnet） | 和主网行为一致但代币无价值，用于部署前的最终验证 |
| RPC 节点 | 你的代码和链之间的通信接口，本地开发用 Anvil，生产环境用 Infura/Alchemy 等服务商 |

### 我的笔记（学完后填写）

**用自己的话解释：为什么需要「本地链」而不直接在测试网上开发？**
一旦在测试网上线，有问题的合约将无法进行更改，或者代价高昂时间成本大，在本地链可以模拟环境，也非常适合测试可能的问题和一些边际极端测试

## 打卡草稿

今日学习： Day 4 学习完成。AI 基础：检索增强生成（RAG）——理解了 RAG 解决 LLM 知识截止和上下文成本问题的思路，学习了 Embedding、向量数据库、Chunk、相关性分数四个核心概念，梳理了离线建库和在线检索的完整流程，以及 RAG Poisoning 这一新攻击面；Web3 基础：开发栈（Dev Stack）——建立了 Solidity → Foundry/Hardhat → Anvil → ethers.js 四层工具链的整体印象，理解了本地链和测试网在开发工作流中的不同定位。

收获 / 思考： 今天最大的认知升级是理解了 RAG 的本质：它不是「把更多东西塞进去」，而是把知识管理和语言生成分开——知识库是 Agent 的外部长期记忆，Context Window 是工作记忆，每次只按需取用相关片段。这也让我意识到 RAG Poisoning 的危险之处：攻击者不需要直接攻击 Prompt，只要在知识库里植入一块恶意文档，等它被检索进上下文就能影响 Agent 行为。Web3 这边建立了一个直觉：本地链是可以随意折腾的沙盒，合约有 Bug 直接重置重来；而测试网和主网一样受不可变性约束，部署前必须在本地把所有边界情况测透。
