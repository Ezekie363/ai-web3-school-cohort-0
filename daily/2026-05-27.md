## AI 基础 — 微调（Fine-tuning）（70 分钟）

### 本章核心一句话

> Fine-tuning 不是「模型效果不好就训练一下」——它适合让模型更稳定地学习某种**格式、风格、领域任务或行为模式**，但不适合用来补实时知识、修权限边界或替代评估。

### 学习重点

**1. 第一性原理：Fine-tuning 改的是模型行为分布，不是产品系统边界（10 分钟）**

遇到模型效果不好，应该先问：
- Prompt 不清楚？上下文缺了？检索没拿到正确资料？输出格式没有 schema？
- 是否已经有 eval 证明问题稳定存在？

三个核心原则：
- **先有 eval，再谈 fine-tuning**：否则不知道微调后是真变好还是只是在少数样本上变顺
- **先修数据，再修模型**：坏数据会把坏习惯训练得更稳定
- **别用微调存实时知识**：实时状态、价格、文档更新更适合放在检索或工具里

**2. 五个知识节点（20 分钟）**

| 节点 | 一句话理解 | 适合场景 | 注意点 |
|------|-----------|---------|-------|
| SFT | 监督微调：用输入+期望输出样本让模型学习任务回答方式 | 固定格式输出、特定语气/风格、领域术语、工具调用样式 | 数据质量极敏感——错误/格式不一致/边界不清都会被学进去 |
| LoRA | 低秩适配：只训练较小的适配参数，降低成本和显存需求 | 开源模型微调，资源有限、快速试验特定任务 | 降低实验成本，但不是魔法——任务定义、数据质量、eval 方法仍决定效果 |
| PEFT | 参数高效微调的统称，LoRA 是其中一种 | 模型较大 + 任务范围明确 + 数据量中等 + 需要多版本 adapter 并行试验 | 不一定要重训整个模型，通过较小参数调整即可适配 |
| Dataset | Fine-tuning 的核心资产：训练集 / 验证集 / 测试集 / 回归集 | 四类都要有，切分要清楚 | **不要把测试集拿去训练**——会让评估完全失去意义 |
| Overfitting | 模型把训练数据记得太死，新样本表现变差 | — | 训练样本少/风格单一/轮数过多/验证集和训练集太像，都会触发 |

**3. 微调前的正确顺序（20 分钟）**

```
① 用 eval 证明问题稳定存在
② 检查 prompt / 上下文 / 检索 / schema 是否能解决
③ 修数据质量
④ 只有 ①②③ 都做了还不够，才考虑 fine-tuning
⑤ 微调后再用同一套 eval 验证是否真的变好
```

**4. 在 AI × Web3 里的位置（20 分钟）**

| Fine-tuning 适合 | Fine-tuning 不适合替代 |
|-----------------|----------------------|
| 交易解释风格 | 链上实时状态查询 |
| 治理摘要格式 | 合约安全审计 |
| 风险标签输出 | 交易模拟 |
| 合约注释风格 | 钱包权限控制 |
| 工具调用样式 | 用户确认 / 外部事实引用 |

Fine-tuning 可以让模型更懂你的任务格式，**但不能让模型直接变成可信执行层**。

### 我的笔记（学完后填写）

**Handbook 说「先有 eval，再谈 fine-tuning」——如果跳过 eval 直接微调，会有什么具体风险？**
相当于不看event直接上手修bug，修不修对位置先不说，还容易导致原来没问题的地方给改了

> **补充（Claude）：** 类比很准。更精确地说有三个具体风险：① **不知道有没有变好**——微调后你没有基准，「感觉顺了」可能只是在训练样本上过拟合了；② **不知道哪里变差了**——可能修了 A 坏了 B，但因为没有 regression set，发现不了；③ **调试难度倍增**——同时改了模型权重和行为，问题到底来自数据、来自微调方法还是来自原始模型，很难拆分排查。Handbook 的逻辑是：先用 eval 固定问题，再用 eval 验证改动，微调只是改动的一种方式。

**Fine-tuning 和 RAG 分别适合解决什么类型的问题？什么时候该用哪个？**
这个问题不清楚

> **补充（Claude）：** 两者解决的是不同维度的问题——**RAG 解决「知识问题」，Fine-tuning 解决「行为问题」**。RAG 适合：模型不知道某些事实，但这些知识可以被检索到（最新资讯、内部文档、链上数据）。Fine-tuning 适合：模型知道怎么回答，但需要以特定格式/风格/流程输出（固定 JSON 结构、特定语气、领域术语习惯）。简单判断：如果问题是「答案在哪里，模型不知道」→ 用 RAG；如果问题是「模型知道答案，但回答方式不对」→ 考虑 Fine-tuning。很多时候两者同时用：RAG 提供知识，Fine-tuning 保证输出格式稳定。

## Web3 基础 — 安全（Security）（20 分钟）

### 本章核心一句话

> Web3 安全不是上线前找人审一次代码，而是从合约设计、权限、测试、交易模拟、监控、应急暂停到权限撤销的**一整套工程流程**。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- 链上系统默认暴露在**公开对抗环境**里，任何可调用路径都要按攻击面看待
- 普通后端有权限隔离、人工回滚、数据库修复；合约没有——代码公开、状态公开、资金公开、攻击者可以反复模拟和抢跑
- 三条底线：**权限必须最小化 / 执行前要模拟 / 上线后要监控**

**五个核心概念：**

| 概念 | 一句话理解 | 关键防护 |
|------|-----------|---------|
| Reentrancy | 合约在外部调用未完成前被再次调用，余额归零前被重复提款 | Checks-Effects-Interactions 顺序 + reentrancy guard；测试多合约交互 |
| Access Control | 谁能执行敏感动作（mint/pause/upgrade/withdraw/setOracle） | 不只问「有没有 onlyOwner」——owner 是 EOA 还是多签？有没有 timelock？私钥泄漏的最坏结果是什么？ |
| Audit | 外部安全审查——不是安全保证书，是发现问题的工具 | 看清审计覆盖了哪些 commit、哪些问题已修复、上线代码是否和审计代码一致 |
| Simulation | 交易发出前的预演：调用哪个合约、转出哪些 token、是否 revert、是否触发授权变化 | 不能替代审计（真实链状态可能在打包前变化），但能挡住链 ID 错/地址错/授权异常等明显错误 |
| Monitoring | 上线后的安全感知层：大额转账、权限变更、预言机异常、TVL 快速流出 | 监控本身不够——必须配「响应」：谁收到告警，谁能 pause，恢复流程是什么 |

**AI × Web3 安全分工：**
```
模型建议 → 工具返回事实 → policy 限制权限
→ simulation 预演结果 → human check 确认高风险 → monitoring 记录执行后果
```

### 我的笔记（学完后填写）

**Handbook 说「安全不是只属于 auditor 的工作」——在 AI×Web3 产品里，哪些环节会引入安全风险？**
还是模型带来的不确定风险，例如prompt injection之类的，模型给的建议需要重新做audi才行

> **补充（Claude）：** 模型风险只是其中一层。Handbook 提到的是**每个角色/环节都会引入风险**，可以按层列出：① **前端**：一个「无限 approve」按钮、一个没有滑点上限的 swap 界面；② **合约**：权限设计过宽、Reentrancy 漏洞、升级权限没有 timelock；③ **AI Agent**：不受限的工具权限、被 Prompt Injection 劫持、生成了危险交易没有 simulation 拦截；④ **运营**：管理员私钥用 EOA 而不是多签、没有监控告警；⑤ **审计**：上线代码和审计版本不一致、部署参数没被审计覆盖。安全是所有这些层的交集，任何一层出问题都会成为入口。
