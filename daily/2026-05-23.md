## AI 基础 — 框架（Frameworks）（70 分钟）

### 本章核心一句话

> AI Framework 不是为了让你少写几行 API 调用，而是把模型、工具、状态、检索、评估和部署组织成一个**可维护的系统**。框架选错，问题通常不是「跑不起来」，而是后面**调不动、测不了、换不掉**。

### 学习重点

**1. 第一性原理：框架是系统边界的表达，不是智能本身（10 分钟）**
- 先引入框架、再让产品逻辑迁就框架——这是很多 AI 项目失败的原因
- 更稳的方式：先画清楚输入、状态、工具、输出、评估和失败路径，再看哪些部分值得交给框架
- 三个判断原则：
  - **简单链路先保持简单**：一次模型调用 + 检索 + 格式化输出，不一定需要复杂 Agent 框架
  - **长流程需要显式状态**：多步任务、工具调用、人工确认、失败恢复，应有可查询的 state
  - **框架要能退出**：难以换模型、换向量库、换部署方式的框架，长期成本很高

**2. 六个知识节点（20 分钟）**

| 框架 | 定位 | 适用场景 | 注意点 |
|------|------|---------|--------|
| LangChain | 组件库：模型接入、prompt、工具、retriever、agent | 快速把模型能力和外部系统接起来；学习 AI 应用常见组件 | 小心「抽象太早」——工作流没想清楚就套框架，排查问题会变难 |
| LangGraph | 工作流 + 状态机：节点执行动作、边控制流、state 记录过程 | 多轮工具调用、重试、人工确认、分支、恢复、长期运行 | 任务只是「问一次答一次」时太重；开始关心任务走到哪步时就该考虑 |
| OpenAI Agents SDK | Agent 工作流工程化：工具、handoff、guardrails、tracing | 组织多 Agent 协作、结构化输出、运行时追踪 | SDK 帮你执行流程，但你要定义工具范围、确认节点、失败标准 |
| DSPy | 可优化的 LM pipeline：用 optimizer 改进 prompt 或调用策略 | 有明确数据集、评估指标和可重复任务（分类、抽取、问答） | 没有 eval 就自动优化，很容易优化到错误目标 |
| Hermes | 面向工具调用和结构化输出的模型 / agent 生态 | 需要稳定 tool calling 和 JSON 输出的场景 | 不要只看榜单分数，要看能否稳定产生需要的结构化格式 |
| Learning Agent | 从反馈、日志、评估结果或用户修正中改进系统 | 有积累数据、完善评估闭环的产品阶段 | 线上反馈直接变行为会引入数据污染；必须先进评估闭环再进生产 |

**3. Learning Agent 的稳健流程（20 分钟）**

```
记录失败样本 → 人工或规则标注原因 → 加入 eval / regression set
→ 修改 prompt / 检索 / 工具 / 模型配置 → 通过测试后再上线
```

关键原则：**学习能力要先进评估闭环，再进入生产系统。**

**4. 在 AI × Web3 里的位置（20 分钟）**

三层分工：

| 层级 | 职责 |
|------|------|
| AI Framework | 管理 prompt、tools、state、eval、trace |
| Web3 基础设施 | 管理账户、签名、合约、交易、链上状态 |
| 产品层 | 定义用户目标、权限边界、确认流程、失败处理 |

框架可以组织 Agent，**不能替用户承担资产风险**。

### 我的笔记（学完后填写）

**Handbook 说「先理解工作流，再决定用不用框架」——你怎么理解？什么情况下你会选择不用框架？**
要对agent在哪个位置进行什么样的工作有清晰的认识，不是盲目的使用，例如一些小的功能实现就可以直接调用api，不需要冗余的框架设计，但是如果要使用框架就要理清工作流

**LangChain 和 LangGraph 的核心区别是什么？如果你要做一个需要多步骤、可恢复的 DeFi 分析 Agent，你会选哪个？**
一个是组件库类型的设计，一个是工作流，工作进行到哪一步的清晰分明，多步骤可恢复首先考虑langraph

**还不清楚的问题：**
AI framework是实际上是对agent、ai开发设计一套功能完善的框架来进行更好的输出合适的内容，是不是这就是工程学上的设计

> **补充（Claude）：** 是的，本质上是软件工程的**关注点分离（Separation of Concerns）**。Framework 把 prompt 管理、工具调度、状态追踪、失败重试这些横切关注点从业务逻辑里剥离出来，让你专注定义产品价值（目标是什么、工具有哪些、失败怎么处理），而不用每次都自己实现一遍基础设施。这和后端框架（Spring、Rails）解决的是同一类问题——只是领域换成了 LLM 应用。

## Web3 基础 — 账户抽象（Account Abstraction）（20 分钟）

### 本章核心一句话

> 账户抽象把「账户如何验证操作、谁来付 gas、哪些权限可以自动执行」从固定的 EOA 模式里释放出来——让钱包更像**可编程账户系统**。

### 学习重点（轻量阅读，建立直觉）

**为什么 EOA 不够用？**
- 必须保管私钥、必须用原生代币付 gas、签名权限很粗、自动化难、恢复体验差
- 账户抽象的核心不是「免 gas」，而是把控制权从**单一私钥**扩展成**可编程规则**

**五个核心概念：**

| 概念 | 一句话理解 |
|------|-----------|
| ERC-4337 | 以太坊账户抽象标准：用户创建 UserOperation，Bundler 打包，EntryPoint 验证执行 |
| Smart Account | 合约控制的账户，可自定义多签、社交恢复、批量交易、限额等策略；合约 bug 也会变成账户风险 |
| Bundler | 收集 UserOperation、模拟验证并提交到 EntryPoint；Bundler 不稳定，用户操作就会卡住 |
| Paymaster | 允许第三方赞助 gas，或用非原生资产付费——新用户无 ETH 也能完成操作；需要风控防止 spam |
| Session Key | 给 Agent 或应用的**临时受限权限**：指定合约、方法、额度、时间、链，到期自动失效 |

**ERC-4337 执行流程：**
```
用户/应用生成 UserOperation
→ 智能账户验证签名、nonce、策略
→ Bundler 打包并提交
→ EntryPoint 调用账户执行目标动作
→ Paymaster 可选择赞助 gas
```

**对 AI Agent 的意义：**
- Session Key 是 Agent Wallet 的关键基础——让 Agent 在受限范围内自动执行低风险动作，高风险仍需用户确认
- 越自动化，越需要清楚的 policy：能调用什么、额度多少、多久过期、谁能撤销、日志在哪

### 我的笔记（学完后填写）

**用自己的话解释：Session Key 和 EOA 私钥有什么本质区别？为什么说它是「AI Agent 权限控制的关键基础」？**
Session是部分放权，EOA私钥是整个账户的操作权，权力的范围不一样，因为AIagent所进行的操作目前来看不一定都是对的，有风险，为了避免损失大量甚至全部资金，Session key是很好的过渡解决办法

> **补充（Claude）：** 理解正确。可以再精确一层：EOA 私钥是**永久的、全范围的、不可撤销的**（泄露就是灾难）；Session Key 是**有期限的、有范围的、可随时撤销的**。这不只是「过渡办法」，而是任何自动化系统的正确架构原则——最小权限（Principle of Least Privilege）。给 Agent 完整私钥，风险不仅来自 Agent 出错，还来自 Prompt Injection、工具漏洞、上下文污染等所有可能让 Agent 被劫持的路径。Session Key 把爆炸半径限制在一个可预期的范围内。
