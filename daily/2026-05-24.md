## AI 基础 — 氛围编程（Vibe Coding）（70 分钟）

### 本章核心一句话

> Vibe Coding 不是「把需求丢给 AI 然后等代码出现」，而是人和 AI Coding Agent **共同迭代**软件：人负责方向、约束和验收，Agent 负责生成、修改、搜索和执行一部分工程动作。

### 学习重点

**1. 第一性原理：把工程反馈循环压短（10 分钟）**
- Agent 可以并行承担搜索文件、改代码、跑测试、看错误、再修改等步骤
- 反馈循环里没有测试、审查和版本控制，速度只会放大混乱
- 三个核心原则：
  - **任务要小**：越具体、越有边界，Agent 越容易产出可审查结果
  - **上下文要准**：让 Agent 看对文件、设计约束和错误输出，比写长篇需求更重要
  - **验证要硬**：运行测试、类型检查、构建、日志，比「看起来对」可靠

**2. 六个知识节点（20 分钟）**

| 节点 | 定位 | 适合 / 不适合 | 关键边界 |
|------|------|--------------|---------|
| Vibe Coding | 人与 AI Agent 共同迭代工作方式 | ✅ 搭原型、修小 bug、补测试、写脚本、重构局部模块 / ❌ 无边界「重写整个项目」 | 范围越大，Agent 越容易改到不该改的地方 |
| Claude Code / Codex CLI | AI Agent 接入本地 repo、终端和测试流程 | 读文件、编辑文件、运行命令、理解测试输出 | 需明确：哪些文件可改、哪些命令可跑、何时必须停下让人确认 |
| Model / API Config | 模型选择、API key、工具权限、成本和日志管理 | 团队可复用的配置方案 | 防止：API key 泄漏到日志/提交、上下文过长成本失控、模型权限不一致 |
| GitHub / gh | AI 生成改动放回版本控制、issue、PR、review 流程 | 让 Agent 多做局部 patch，少做不可追踪的大改动 | 每次改动后检查：git diff、修改文件列表、测试结果、是否含密钥或构建产物 |
| CLI / Script | Agent 生成脚本、迁移脚本、批量修改脚本 | 一次性脚本、数据检查、批量修改 | 读写范围确认 + 会改外部状态的脚本必须先 dry run |
| Repo Workflow | 把 AI Coding 接进完整工程流程 | issue → branch → commit → test → review → merge | 不要让 AI 绕过流程，让它参与流程（生成 patch、跑测试、写 PR summary） |

**3. Repo Workflow 中 Agent 的参与方式（20 分钟）**

```
从 issue 提炼任务边界
→ 搜索相关文件
→ 生成最小 patch
→ 跑测试并解释失败
→ 补充 changelog 或文档
→ 写 PR summary 和验证说明
```

AI Coding 的质量，最终还是要落到 repo workflow 里。

**4. 在 AI × Web3 里的位置（20 分钟）**
- Vibe Coding 显著加快探索速度，尤其适合 hackathon 和早期原型
- 链上代码风险更高：**合约、签名、权限、支付、迁移脚本不能只靠 Agent 生成后直接上线**
- AI Coding 可以帮你写测试、解释 ABI、生成脚本，但高风险动作必须经过审查、模拟和多方确认

### 我的笔记（学完后填写）

**Handbook 说「AI 能帮你写更多代码，但不能替你承担工程判断」——结合你用 Claude Code 的经历，你觉得哪些判断必须由人来做？**
首先最重要的就是关于细节的敲定和一些权限放权的问题，功能的细节落实和技术栈必须由人来进行审核和判断，这样会大大减少后续不必要的review以及修改的时间，还有就是权限的问题，不能给ai无限制的发挥，不仅效果可能不符合预期，还有安全风险

> **补充（Claude）：** 可以再分两层来记：「做什么」永远是人的判断（目标、边界、约束、验收标准）；「怎么做」Agent 可以大量参与，但结果必须由人审查。Handbook 里说的「上下文要准」也是工程判断的一部分——你决定让 Agent 看哪些文件、哪些约束，就决定了它能做出多好的判断。给错上下文，Agent 会生成出技术上正确但方向完全错误的代码。

**为什么链上相关代码（合约、签名、权限）比普通业务代码需要更严格的审查，即使是 AI 生成的也一样？**
因为涉及上链的东西，一旦上链无法更改，稍微的权限安全问题就有可能瞬间导致资金全失（黑暗森林法则），ai也一样，同样需要合约安全审计（openevmbench）

> **补充（Claude）：** 黑暗森林法则用在这里很准——链上所有公开的合约代码和状态都在被机器人实时扫描，漏洞通常在几分钟内就会被利用，没有「发现了再打补丁」的窗口。AI 生成的代码会继承模型训练数据里的常见模式，但安全漏洞往往藏在边缘情况和模式之外——这也是为什么合约审计必须是独立的人工或专业工具来做，而不能只靠 AI 自我检查。

## Web3 基础 — 去中心化金融（DeFi）（20 分钟）

### 本章核心一句话

> DeFi 是建立在智能合约上的开放金融系统——它让资产、交易、借贷、稳定币和流动性可以被**协议化组合**，也让风险沿着**协议依赖链传播**。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- DeFi 协议管理的是**资产状态**，不只是应用界面——余额、抵押、债务、流动性、清算全在合约里
- 资产是协议输入：token 标准、decimals、授权和余额都会影响协议行为
- 流动性决定可执行性：有价格 ≠ 能成交，能成交 ≠ 低成本成交
- 可组合性放大风险：一个协议出问题，可能影响依赖它的多个协议

**五个核心概念：**

| 概念 | 一句话理解 | 关键风险 |
|------|-----------|---------|
| Token | DeFi 的资产单位，ERC-20 最常见；不同 token 标准、精度、权限和风险差异很大 | 错误的合约地址和 decimals 处理是常见风险源 |
| AMM | 用流动性池和定价公式替代订单簿，价格由池子状态决定 | 滑点、无常损失、价格影响、MEV（sandwich attack） |
| Lending | 存款赚利息 / 抵押借款，健康因子低于阈值触发清算 | 抵押品价格下跌 + 预言机延迟 + 流动性不足 = 清算失败 |
| Stablecoin | DeFi 的计价和结算基础；「稳定」来自不同机制，不是天然保证 | 脱锚时谁承担损失？储备在哪里？流动性有多深？ |
| Liquidity | 决定资产能否以合理价格买卖、借出、清算或退出 | 流动性不足 → 滑点大 / 清算困难 / 预言机易被操纵 |

**AI Agent 执行 DeFi 操作的最低要求：**
额度限制、协议白名单、滑点上限、模拟结果、预言机检查、人工确认、交易后审计记录

### 我的笔记（学完后填写）

**Handbook 说「有价格不等于能成交，能成交不等于低成本成交」——用一个具体 DeFi 场景解释这句话：**
最常见的uniswap的例子，一个swappool里面有价格，但是假设流动性提供者很少，如果你要兑换的金额超过提供的这部分，就会导致价格无法正常成交，假设能成交，也可能导致价格滑点严重，也就是你兑换的资产缩水了

> **补充（Claude）：** 举例准确。可以再加一层「能成交不等于低成本」的维度——即使流动性够、滑点可接受，MEV 机器人可能在你的交易前后各插一笔（sandwich attack）：先把价格拉高，等你的交易成交后再把价格压回去，把你的滑点损失套走。所以同一笔 swap，在链上「能成交」和「以你预期的价格成交」是两件事。这也是为什么 Agent 执行 DeFi 操作必须设置滑点上限——不只是防止流动性不足，也是防止 MEV。
