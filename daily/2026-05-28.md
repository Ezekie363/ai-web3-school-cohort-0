## AI 基础 — 推理服务（Inference）（70 分钟）

### 本章核心一句话

> 训练决定模型学到了什么，推理决定模型在真实产品里如何响应用户——对 builder 来说，Inference 是**延迟、成本、上下文、稳定性和部署边界**的综合选择，不是一个云服务按钮。

### 学习重点

**1. 第一性原理：在约束条件下交付可用答案（10 分钟）**
- 同一个模型在 API、GPU 服务、本地量化和边缘设备上表现完全不同
- 三个核心判断：
  - **质量有代价**：更强模型通常意味着更高成本、更长延迟或更复杂部署
  - **部署改变边界**：API model 少管基础设施；本地模型多拿控制权，也多承担运维
  - **服务要可替换**：把模型调用封装清楚，才能做 fallback、灰度和评估

**2. 四个知识节点（20 分钟）**

| 节点 | 一句话理解 | 优势 | 关键问题 |
|------|-----------|------|---------|
| API Model | 通过云端服务调用模型，上手快、模型更新快、基础设施负担低 | 快速上线，模型能力强 | 速率限制、超时、重试、日志脱敏、账单控制、版本变更——Agent 多轮调用会放大成本和延迟 |
| Local Model | 在私有环境运行模型，可控性强 | 隐私要求高、成本敏感、离线运行、深度定制 | 显存、量化方式、推理框架、并发能力都会影响效果；通常承担特定任务而非替代所有 API |
| Quantization | 把模型权重精度降低（FP16→INT8/INT4），减少显存需求 | 让个人电脑或小 GPU 运行大模型 | 可能降低输出质量，尤其是长推理、代码生成、工具调用场景——必须用任务样本测试而非主观感受 |
| Serving | 把模型部署成稳定服务：并发、队列、流式输出、缓存、监控、失败处理 | 支撑真实产品流量 | 请求失败如何重试降级？模型版本如何灰度？日志如何脱敏？成本延迟错误率如何监控？ |

**3. 推理服务的核心问题清单（20 分钟）**

```
□ 请求失败时怎么重试或降级？
□ 模型版本怎么灰度？
□ 输入输出日志如何脱敏？
□ 长请求是否要进入队列？
□ 成本、延迟和错误率如何监控？
□ Agent 多轮调用时成本如何控制？
```

**4. 在 AI × Web3 里的位置（20 分钟）**
- 链上动作不可逆——推理层必须留下可审计记录：用了哪个模型、输入来自哪里、输出是什么、是否触发工具、失败时如何处理
- 不能像普通聊天一样随意——模型参与交易解释、合约分析、策略建议时，推理层的稳定性直接影响安全性

### 我的笔记（学完后填写）

**Handbook 说「服务要可替换」——什么是 fallback？为什么推理层需要设计 fallback 而不是只依赖一个模型？**
fallback就是备选方案，你的模型可以是api，或者本地，或者其他的模型，因为一旦你的模型服务挂机会影响你的产品使用，这个时候你就需要备选方案

> **补充（Claude）：** 方向正确。补充两点让 fallback 概念更完整：① **fallback 的层次**——不只是「模型 A 挂了换模型 B」，还包括：API 超时 → 重试同一模型；重试失败 → 降级到更小/更快的模型；全部失败 → 返回安全的默认响应而不是报错崩溃。② **为什么要把调用封装清楚**——Handbook 说的「服务要可替换」，核心是代码层隔离：你的业务逻辑不直接调用某个模型 API，而是调用一个统一的推理接口，背后换 GPT、换 Claude、换本地模型都不影响上层代码。这样做了才能在不改业务逻辑的前提下做灰度（部分流量走新模型）、做评估（同一请求同时打两个模型对比）、做 fallback（主模型失败自动切备用）。在 AI×Web3 里这点更重要——链上动作失败代价高，推理层必须能预期失败并有处理方案，而不是让调用方猜。

**API Model 和 Local Model 各有什么不可替代的场景？在 AI×Web3 里什么情况下必须用 Local Model？**
API直接帮你包装好，你几乎不需要任何的基础设施即可使用，local的优点是稳定性，不会挂，但是速度，显存，上下文等都是问题，alxweb3里像一些涉及隐私的就可以考虑local

> **补充（Claude）：** 理解方向正确，但「local 更稳定不会挂」这点要谨慎——本地模型是你自己维护，硬件故障、显存溢出、版本冲突都会导致它挂，只是「不依赖外部服务商」而非「一定更稳定」。更准确的分法是看**控制权和数据边界**：API Model 的不可替代场景是：最强能力需求（复杂推理、代码生成）、快速原型、不自己维护基础设施的小团队；Local Model 的不可替代场景是：① **合规要求**——金融、医疗等行业数据不能离开私有环境；② **链上私钥相关**——私钥、助记词、签名内容绝对不能发给第三方 API；③ **高频低延迟任务**——链上价格监控、实时预警，API 的网络延迟和 rate limit 会成为瓶颈；④ **离线或边缘设备**——硬件钱包、离线签名环境。在 AI×Web3 里，凡是上下文里含有私钥、未签名交易或用户敏感资产信息的场景，必须用 Local Model 或至少要保证输入在发给 API 前做了严格脱敏。

**还不清楚的问题：**
不是很理解Quantization和serving，这两者是什么意思？

> **补充（Claude）：** **Quantization（量化）**：模型的权重（参数）本来是用高精度数字存储的，比如 FP16（16位浮点数）。量化就是把这些数字压缩成更低精度，比如 INT8（8位整数）甚至 INT4（4位整数）。类比：原来存的是「3.14159265」，量化后存「3.14」——数字变小了，占的空间和运算量也少了，一块普通显卡或者个人电脑就能跑起来。代价是精度损失——量化越激进，模型回答可能越不准确，尤其在复杂推理和工具调用时更明显。**Serving（服务化）**：模型本身只是一堆权重文件，没法直接接收用户请求。Serving 就是把模型包装成一个稳定运行的服务——处理并发（同时有多个用户请求怎么办）、管理队列（请求太多时怎么排队）、支持流式输出（边生成边返回）、做缓存（同样的请求不重复跑）、监控健康状态（模型是否还在运行）、处理失败（超时重试还是直接报错）。简单说：Quantization 解决「模型能不能跑起来」，Serving 解决「跑起来之后能不能撑住真实流量」。

## AI × Web3 Bridge — 链感知上下文（Chain-aware Context）（20 分钟）

### 本章核心一句话

> 模型不能靠语言记忆判断链上事实——Chain-aware Context 把链上状态（地址、余额、交易、授权、ABI、事件）变成模型可读、可引用、可验证的上下文包，是所有链上 Agent 行动的事实基础。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- 模型知道「Uniswap 是 DEX」没用——真正执行时需要具体网络、合约、池子、用户余额、allowance、滑点和交易模拟结果
- **链上状态有时间性**：同一地址的余额、授权、仓位会随区块变化，上下文要带时间戳和数据来源
- **事实和解释要分开**：工具返回事实，模型负责解释——不能把模型猜测当链上事实

**七个知识节点：**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| On-chain Data | 初级 | 链上可直接验证的数据：余额、交易、日志、合约状态、区块信息 | 读取时必须带：chain id、block number、contract address、method、返回值、读取时间 |
| Contract Docs | 初级 | 帮模型理解合约设计意图、参数含义、权限边界 | ABI 只有函数签名，文档补足语义——但文档可能过期，要用链上数据交叉验证 |
| ABI / Event | 中级 | Agent 理解合约可调用能力和历史行为的核心结构 | **能调用 ≠ 应该调用**：写交易前还需权限、余额、allowance、simulation、policy 检查 |
| Transaction History | 中级 | 理解用户或合约过去做过什么 | 必须保留：tx hash、block number、from/to、method、value、token transfers、logs——不能只看自然语言总结 |
| Explorer Context | 初级 | 区块浏览器提供的可视化链上证据 | 给 explorer link 比只说「交易成功」更可靠——用户可自行核验，开发者可排查错误 |
| Indexing Context | 中级 | 把链上事件整理成面向产品查询的数据 | 必须带时间戳和同步状态——落后 500 个区块的索引不能被当作当前事实 |
| Citation | 初级 | 让模型回答能回到具体链上证据或文档来源 | 可以是：tx hash、block number、合约地址、event log、explorer 链接——没有 citation 的链上解释只是观点 |

**在 AI × Web3 里的位置：**
- Chain-aware Context 是所有链上 Agent 的**输入层**——没有这层，Web3 Tool Use、Agent Workflow、Agent Wallet 都建立在不可靠上下文上
- 一个好的链感知上下文包应包含：用户目标、chain id 和网络名称、用户地址和余额、相关合约地址/ABI/文档/风险提示、最近交易和授权、索引数据更新时间、每条关键结论的 citation

### 我的笔记（学完后填写）

**Handbook 说「上下文要区分事实和解释」——如果把模型的解释当成事实传递下去，会出现什么问题？**
出现地址错误、价格错误等等幻觉让人导致以为是真实的数据信息，做出错误的判断和授权

> **补充（Claude）：** 方向正确。更关键的问题是**误差会在 Agent 的执行链路里逐步放大**——模型把一个合约地址猜错了，如果这个「解释」被当成事实传给下一步工具调用，工具就会用错误地址去查 ABI；拿到错误 ABI 再生成交易草稿，草稿里的参数编码就是错的；用户看到的 UI 展示也是错的，如果没有严格 simulation，这笔交易打出去就是真实损失。普通 AI 应用里解释错了顶多是答案不准，用户重问一次就好；链上系统里解释错了会传导成执行错误，而且链上不可撤回。Handbook 强调「工具返回事实，模型负责解释」，本质上是要求：**每一条被传入下一步的数据，都必须能溯源到工具调用结果，而不是模型的生成内容**。

**Citation 在普通 AI 应用里不是必须的，但在 AI × Web3 里为什么是必须的？**
因为涉及到真正的钱，不得马虎，一旦失误，用户的资金将损失

> **补充（Claude）：** 正确，补充两个更具体的理由：① **链上有客观真相可以对照**——普通 AI 应用说「今天天气不错」，没有办法验证；但链上 AI 说「这个地址的余额是 2.5 ETH」，有具体的区块和状态可以对照。Citation（附上 block number、tx hash、explorer 链接）让用户能直接去链上核验，而不是只能相信 AI。② **责任可追溯**——如果 Agent 建议了一个有问题的操作，「AI 说的」不是免责理由；但如果有 citation，开发者能追查是数据源的问题、是模型解读的问题还是参数的问题，才能修复和改进。没有 citation 的链上解释，出了问题连原因都找不到。Handbook 说得准：「带 citation 的解释才有机会被验证和追责」——验证是为了用户安全，追责是为了系统可改进。
