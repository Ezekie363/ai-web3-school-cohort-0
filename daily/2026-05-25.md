## AI 基础 — 模型上下文协议（MCP）（70 分钟）

### 本章核心一句话

> MCP 解决的不是「模型会不会更聪明」，而是「模型如何以**可描述、可复用、可管理**的方式使用外部能力」——它是 AI 应用的工具接口标准。

### 学习重点

**1. 第一性原理：模型通过明确协议访问被授权的上下文和工具（10 分钟）**
- 每个应用都用自己方式接工具 → Agent 生态碎片化，权限/schema/日志难统一
- MCP 的关键不是「能接更多工具」，而是让工具接入**可描述、可发现、可限制**
- 三个核心原则：
  - **工具要有 schema**：没有 schema，模型调用工具就会变成自然语言猜参数
  - **权限要在协议外也成立**：协议能描述能力，但真正的授权、审计和隔离仍要由系统实现
  - **错误要可传递**：工具失败、超时、权限不足，必须明确返回，不能让模型猜

**2. 四个知识节点（20 分钟）**

| 节点 | 职责 | 设计重点 | 风险点 |
|------|------|---------|-------|
| Server | 提供能力：暴露 resources、tools、prompts | 明确边界：哪些只读、哪些有副作用、参数 schema、错误返回、日志 | 不加限制的 server 会把本地文件、内部 API 或高风险动作直接暴露给模型 |
| Client | 连接模型和 server：发现能力、转发调用、展示工具 | 用户知情：当前连了哪些 server、模型能调用哪些工具、调用前是否需要确认 | client 不做权限提示，用户完全不知道模型在做什么 |
| Tool Schema | 描述工具名字、用途、参数、返回值和约束 | 说清楚：什么时候用、参数代表什么、是否会修改外部状态、失败返回什么 | schema 写得模糊 → 模型用错误参数填空 |
| Permission | 区分动作风险等级，决定授权粒度 | 至少区分：只读/写入、会话/长期授权、是否需要确认、是否可撤销和审计 | 读文档和删文件的风险完全不同，用同一权限模型是最常见的设计错误 |

**3. Tool Schema 的完整要素（20 分钟）**

一个好的 schema 必须包含：
- 这个工具**什么时候用**
- 参数**代表什么**（不只是字段类型）
- 哪些字段**必填**
- 是否会**修改外部状态**（有无副作用）
- **失败时**返回什么

**4. 在 AI × Web3 里的位置（20 分钟）**

稳定的分工：

| 层级 | 职责 |
|------|------|
| MCP | 工具发现和调用格式：读区块浏览器、查合约文档、调用 RPC、生成交易草稿 |
| Web3 账户系统 | 最终权限和执行边界：账户权限、交易模拟、签名确认、session key、审计日志 |

MCP 本身**不是钱包安全方案**——它标准化工具入口，不能替代 Web3 侧的授权和执行控制。

### 我的笔记（学完后填写）

**Handbook 说「权限要在协议外也成立」——MCP 协议本身能描述什么、但不能保证什么？**
mcp协议本身能描述对工具的使用方法和一些可访问的数据源，我觉得不能保证的是，这个工具本身是否有问题（不确定）

> **补充（Claude）：** 方向正确。更精确地说：MCP 协议能描述工具的 schema（输入/输出/用途），但不能保证——① 工具真的只做它声明的事（实际执行仍在 server 里，协议只是描述层）；② 调用被真正授权（协议传递授权请求，但审计和执行是系统的责任）；③ 工具之间有隔离（一个 server 里多个工具可能共享权限）；④ 调用记录被正确留存（日志是系统实现，不是协议保证）。简单说：协议描述「能做什么」，系统才能保证「只做被允许的事」。

**结合你用 Claude Code 的经历（它本质上是一个 MCP Client），「让用户知道模型能调用哪些工具」这件事为什么重要？**
直接关乎产出的的项目的质量，好的mcp接入结合适合的项目会产生事半功倍的效果，我认为本质上是好马配好鞍的比喻，mcp要接入合适的、安全的才能更好的使Claude code发挥更大的作用

> **补充（Claude）：** 你说的是工具选型质量，这也对。但 Handbook 这里更强调的是**安全透明度**：如果用户不知道模型能调用哪些工具，就无法发现异常（比如模型调用了一个本不该用的工具）、无法做知情同意（你以为只是问答，实际上模型在写文件）、也无法事后审计。Claude Code 每次调用工具都会展示给你确认，就是这个原则的体现——不是为了限制速度，而是让你始终知道 Agent 在做什么，随时可以叫停。

**还不清楚的问题：**
mcp跟skill的本质区别是什么？

> **补充（Claude）：** 很好的问题，两者在不同层级工作：**MCP 是工具层**——它定义模型能调用的外部能力（函数调用、API、文件操作），以 JSON-RPC 协议运行，是运行时动态发现和执行的；**Skill 是指令层**——它通过 prompt/上下文告诉模型「遇到这类任务应该怎么做」，是自然语言的工作流和规程，在模型推理时作为指导。类比：MCP 给模型「手」（能做的动作），Skill 给模型「脑中的方法论」（如何思考和规划）。两者可以组合：Skill 告诉模型什么时候该用什么工具，MCP 提供实际执行那个工具的接口。

## Web3 基础 — 预言机（Oracle）（20 分钟）

### 本章核心一句话

> 区块链自己不能天然知道链外世界发生了什么——Oracle 把价格、储备证明、随机数、事件结果等外部信息以**可被合约使用的方式**带进链上，同时也把链外风险带进链上。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- 外部数据一旦进入链上，就会变成**协议规则的一部分**——feed 的更新频率、数据源、聚合方式都会影响用户资产安全
- 数据源是信任边界：谁提供数据、谁聚合数据、谁能更新数据，都必须清楚
- 延迟会变成风险：旧价格可能导致错误清算或坏账
- 异常要有保护：合约应处理 stale price、极端跳变和 feed 暂停

**四个核心概念：**

| 概念 | 一句话理解 | 关键检查点 |
|------|-----------|-----------|
| Price Feed | 最常见的 Oracle 形式，为合约提供资产价格 | feed 地址是否正确、decimals、更新时间是否过旧、返回值是否异常 |
| Data Feed | 价格之外的链外数据：储备证明、利率、体育结果、随机数 | 数据从哪来、如何更新、谁能提交、是否可验证、出错后怎么处理 |
| Oracle Risk | 链外数据进入链上执行引入的系统性风险 | 数据源被操纵、feed 延迟、聚合节点离线、stale price 未检查、低流动性价格被攻击 |
| AI Oracle | 把模型推理/评分/分类结果提交给链上系统 | 输入可追溯、模型版本记录、输出可复核、允许挑战或仲裁、错误输出的链上后果 |

**AI Agent 使用 Oracle 的合理方式：**
> AI 给出结果 → 系统记录来源和置信度 → 高风险场景引入 human-in-the-loop、挑战期、多人验证或经济惩罚

不是「模型说什么，合约就执行什么」。

### 我的笔记（学完后填写）

**「数据源是信任边界」——如果一个 DeFi 借贷协议使用了被操纵的价格 feed，会发生什么？谁会受损？**
用户资金受损，发生异常清算，相当于市场价格混乱，引发非常多的资金问题，一些正常的借贷，合约都会出现清算风险（假设原本正常的价格，就没有问题）

> **补充（Claude）：** 理解正确，可以再具体一层——典型攻击路径是：攻击者用闪电贷（flash loan）在低流动性市场大量买入某 token，把它的价格在预言机采样窗口内短暂拉高；再把这个被高估的 token 作为抵押品，从借贷协议借出远超实际价值的资产；闪电贷还款后价格恢复，但协议已经损失。受损方是协议流动性提供者（存款人），攻击者套利离场。这也是为什么 Handbook 强调要检查「低流动性资产价格被攻击」这个 Oracle Risk。
