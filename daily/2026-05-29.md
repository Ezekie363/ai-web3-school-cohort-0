## AI × Web3 Bridge — Web3 工具调用（Web3 Tool Use）（70 分钟）

### 本章核心一句话

> Web3 Tool Use 是把 RPC、合约读取、交易生成、钱包确认和 DeFi 操作变成 Agent 可调用工具的过程——真正难的不是「能调用」，而是**读写分离、参数结构化、模拟确认和日志可审计**。

### 学习重点

**1. 第一性原理（10 分钟）**
- **模型可以选择工具，但工具必须用确定性边界限制模型**——不要让 Agent 直接拼接任意 calldata 或调用任意地址，工具应把危险能力封装成受限接口
- **读写分离**：读取链上状态和发送交易必须是不同工具、不同权限——不能混在一个"万能 RPC"工具里
- **参数结构化**：chain id、contract address、method、args、value、slippage 不能埋在自然语言里
- **日志不可省**：每次工具调用都要记录输入、输出、时间、来源和错误

**2. 八个知识节点（20 分钟）**

**只读工具（相对低风险）：**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| RPC Tool | 初级 | 读取链状态、查询区块、估算 gas、获取日志 | 只读与广播必须拆分；返回值必须带：chain id、RPC provider、block number、method、result、error |
| Contract Read | 初级 | 调用合约 view/pure 函数，不改变链上状态 | 常用且相对低风险，但注意：读错网络/合约/ABI 不匹配/RPC 数据滞后都会让模型基于错误事实生成建议 |
| Explorer Tool | 初级 | 查询交易、合约源码、event、token transfer、地址历史 | 提供可验证证据：交易是否成功、调用了什么方法、转了哪些 token、合约是否 verified |

**写入/高权限工具（需严格控制）：**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| Contract Write | 高级 | 改变链上状态，必须经过模拟、权限和确认 | 写前至少需要：chain id + 合约地址 + ABI method/args + value 预估 + gas 估算 + simulation + policy 检查 + 用户授权 + tx hash 追踪；限白名单合约/方法/额度 |
| Wallet Tool | 高级 | 连接账户、请求签名、生成交易、管理授权 | 最敏感边界；connect/签名/发送交易/授权 token/撤销授权必须分开；AI 生成的交易草稿不能绕过钱包确认 |
| DeFi Tool | 高级 | swap、借贷、授权、仓位查询、清算风险的封装 | 必须要求：协议白名单 + 最大交易额 + 最大滑点 + 价格来源 + simulation + allowance 检查 + 人工确认或 session key policy |

**权限与记录：**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| Tool Permission | 高级 | 定义 Agent 能调用哪些工具、在什么条件下调用、传什么参数 | 分层：查询余额→自动允许；小额白名单支付→session key；大额转账/授权→人工确认；任意合约调用→默认禁止 |
| Tool Log | 中级 | Agent 行为可审计的基础 | 每次调用至少记录：用户目标、工具名、输入参数、输出结果、错误、时间、chain id、block number、tx hash、确认人、policy 判断 |

**3. 在 AI × Web3 里的位置（20 分钟）**
- Web3 Tool Use 是从「AI 能解释链上信息」走向「AI 能参与链上执行」的关键层——连接 Chain-aware Context、Agent Workflow 和 Agent Wallet
- 风险随执行能力升级：**只读工具 → 解释错误风险；写链工具 → 资产和权限风险**
- 越靠近执行，越需要 policy、simulation、human check 和日志

### 我的笔记（学完后填写）

**Handbook 说「读写分离：读取和发送交易必须是不同工具」——为什么不能是同一个工具、通过参数区分读写？**
因为同一个工具如果读取的时候出错了，直接影响写交易的部分，读写分离可以降低这个风险

> **补充（Claude）：** 方向正确，但更核心的理由是**权限边界和代码安全**。如果读写在同一个工具里，权限就必须同时包含「读」和「写」——这意味着：① 模型调用这个工具时，无论它选了读还是写的参数，系统都已经授予了写链的权限，权限边界消失了；② 一个 bug 或 Prompt Injection 只需要改参数就能把「查余额」变成「发交易」，不需要突破任何权限层。读写分离后，只读工具在代码层就没有签名和广播能力，即使模型产生幻觉传了错误参数，工具本身也无法发出交易——这是**系统层的硬边界**，不是靠参数检查的软约束。类比：数据库里读写权限是分开的账号，不是同一个账号用参数控制只读还是写入。

**Tool Permission 分层设计里，「过度宽松」和「过度严格」各会带来什么问题？在 AI×Web3 里哪个方向的错误更难接受？**
一定是过渡宽松，因为web3的世界一旦设计交易，智能合约的部分出错，资金将是实打实的损失

> **补充（Claude）：** 正确。补充两边的具体后果让分析更完整：**过度宽松的问题**——Agent 可以调用任意合约、任意方法、任意额度，Prompt Injection 或模型幻觉就能直接触发资产转移；链上不可撤回，损失是永久的。**过度严格的问题**——产品无法使用；用户每一步都需要人工确认，Agent 失去执行价值，沦为只是会解释的聊天机器人；开发团队也会因为频繁被 policy 拦截而绕过检查，反而制造了更大漏洞。所以权限设计的目标是**风险分级而非全面禁止**——根据动作的不可逆程度和影响范围决定自动化程度。Handbook 的分层就是这个逻辑：只读查询自动化没问题，大额转账必须人工，中间的小额白名单交给 session key policy 执行。「更难接受」是过度宽松，因为代价不对称——宽松出错是永久资产损失，严格出错是用户体验差但可以修。

## AI × Web3 Bridge — 智能体工作流（Agent Workflow）（20 分钟）

### 本章核心一句话

> Agent Workflow 的核心是**把概率模型放进确定性流程里**——用户目标 → 上下文读取 → 计划生成 → 工具调用 → 风险检查 → 执行 → 记录复盘，而不是让模型自由发挥。

### 学习重点（轻量阅读，建立直觉）

**第一性原理：**
- 高风险 Agent 不能只有「下一步推理」，必须有**状态、边界和停止条件**
- **流程要显式**：不要把完整执行链路藏在一段长 prompt 里
- **状态要可恢复**：工具失败、用户拒绝、交易 pending 时，系统要知道如何继续或停止
- **评估要可回放**：没有 trace 和 regression set，改模型后不知道是否更安全

**七个知识节点：**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| Task Graph | 中级 | 把目标拆成节点和依赖，每步有确定的输入/输出/权限/停止条件 | 例：swap 拆成 9 步：读上下文 → 查余额 → 查价格 → 生成候选交易 → 模拟 → 展示风险 → 用户确认 → 发送 → 追踪 |
| State Machine | 高级 | 让 Agent 执行过程有明确状态，不只依赖聊天历史 | 常见状态：draft → context_loaded → plan_ready → simulation_failed → waiting_user_confirmation → submitted → confirmed → reverted → cancelled；防止 RPC 失败或重启后重复执行危险动作 |
| Human-in-the-loop | 中级 | 把人放在关键风险点，而不是让人确认每一步 | 分层：只读→自动；草稿→自动；小额白名单→session key；高风险→人工确认；超出 policy→拒绝；重点：用户确认时能否真正看懂资产/权限/失败风险 |
| Retry / Fallback | 中级 | 处理工具失败、网络异常、模型不合格、交易状态不确定 | 读取失败可重试；发送失败先判断是否已广播；pending 不能简单再发一笔；fallback 可降级成只读模式，但不应自动换未经评估的模型继续发交易 |
| Trace | 初级 | Agent 每一步输入、判断、工具调用和结果的完整记录 | 至少包含：用户目标、模型版本、上下文来源、工具 i/o、policy 判断、simulation 结果、人工确认、tx hash、最终状态 |
| Evaluation Harness | 高级 | 系统测试 Agent 在不同任务、风险和异常场景下的表现 | 不只测回答好不好：还要测是否拒绝越权/识别错链错合约/缺数据时停止/要求 human check/避免危险 calldata |
| Regression Set | 中级 | 固定测试用例，防止模型或工具更新后安全性退化 | 应覆盖：正常 swap、错误链、无限 approve、恶意文档诱导、余额不足、oracle 价格过旧、用户拒绝签名、交易 pending 超时 |

**在 AI × Web3 里的位置：**
- Agent Workflow 是 Bridge 层的**流程骨架**：Chain-aware Context 提供事实，Web3 Tool Use 提供能力，Agent Wallet 提供权限边界，Workflow 把它们组织成可执行但可控的路径
- 没有 workflow → 「模型直接调用工具」——demo 快，真实资产面前不够用

### 我的笔记（学完后填写）

**Handbook 说「状态要可恢复」——如果没有 State Machine，当一笔交易处于 pending 状态时 Agent 重启，可能发生什么？**
可能会重新查询这笔交易，这样大大耗费了时间成本和rpc的流量成本，甚至可能重新分析意图进行交易的提交耗费token

> **补充（Claude）：** 方向正确，但最严重的风险比效率问题更危险——**Agent 可能把同一笔交易发出去两次**。如果没有 State Machine，重启后 Agent 不知道「已经提交过一笔 pending 中的交易」，它看到用户的原始目标，重新走一遍流程：生成交易草稿 → simulation 通过 → 用户（或 session key）再次确认 → 第二笔交易发出。结果是两笔相同交易都被打包上链，用户损失双倍资金，或者 nonce 冲突导致一笔 revert。Handbook 说的「交易 pending 不能简单再发一笔」正是这个场景。有 State Machine 时，重启后系统查到状态是 `submitted`，就只做一件事：监听这笔 tx hash 的回执，而不是重新执行。这是「状态」比「聊天历史」更可靠的根本原因——聊天历史可以被截断或遗忘，状态是显式存储的事实。
