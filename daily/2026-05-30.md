## AI × Web3 Bridge — 智能体钱包（Agent Wallet）（90 分钟）

### 本章核心一句话

> Agent Wallet 的第一性原理：**控制权不能交给一个概率系统**——Agent 只能拿到可验证、可限制、可撤销的行动空间，目标是让 Agent 可以执行，但只能在用户预先允许的边界里执行。

### 学习重点

**1. 第一性原理（10 分钟）**

为什么不能直接给 Agent 一个私钥：
- Agent 的判断来自模型、上下文和工具返回值——可能很有用，也可能被错误信息、恶意文档或不完整状态带偏
- 设计重点不是「让 Agent 更像真人钱包用户」，而是把行动范围变成**账户和系统都能检查的规则**

三条核心原则：
- **Agent 不直接控制主资产**：模型可以生成计划和交易草稿，但不应该拿到用户主私钥，高风险动作交回人确认
- **钱包不只是签名按钮，而是权限系统**：表达时间、金额、合约、方法、资产、接收方和撤销条件，让模糊授权变成可检查规则
- **自动化必须绑定撤销能力**：用户随时要能看见 Agent 拿了什么权限、做过什么、还能做什么，以及在哪里关掉它

**2. 九个知识节点（40 分钟）**

| 节点 | 难度 | 一句话理解 | 关键要素 |
|------|------|-----------|---------|
| AA Wallet | 中级 | 基于 Account Abstraction 的钱包——账户终于可以表达规则，而不只是一把私钥控制一个账户 | EOA = 私钥控账户；AA Wallet = 可把签名、权限、gas 支付、多签、限额、session key 等逻辑放进账户规则 |
| Smart Account | 中级 | 「带规则的钱包账户」——执行边界用合约规则承载，不只是保存资产 | 可定义：谁能发起操作、哪些需确认、哪些合约可调用、每日限额、权限何时失效、异常时如何暂停；**「授权给 Agent」必须拆成可检查规则，而不是模糊的「允许自动操作」** |
| Safe | 中级 | 多签和智能账户基础设施——适合团队/DAO 把执行权分散，而不是由单一 Agent 独占控制权 | Agent 可生成草稿或触发低风险模块；大额转账/合约升级仍需多签确认；Agent 参与流程，不独占最终控制权 |
| Session Key | 高级 | 「临时有限权限的小钥匙」——主私钥和零权限之间的平衡点 | 必须限制：有效时间、可调用合约/方法、单笔金额和总额度、可操作 token 类型、是否允许转出外部地址、是否可随时撤销；**Session Key ≠ 小号私钥，= 一组受限能力** |
| Policy | 高级 | 把「Agent 可以做什么」写成系统能检查的规则，而不是自然语言描述 | 例：每天最多 10 USDC、只调用白名单合约、只能 swap 不能无限 approve、滑点超 1% 停止、NFT 转出必须人工确认；关键：系统能否回答「这笔操作有没有越界？」 |
| Guard | 高级 | 执行前的确定性拦截层——Agent 生成候选动作，Guard 负责拒绝越界动作 | 检查：目标地址是否白名单、方法是否允许、金额是否超限、授权额度是否异常、simulation 是否符合预期、市场状态是否已变化 |
| Simulation | 中级 | 签名前预演交易结果——不只是技术检查，也是用户理解和确认的入口 | 要翻译成人能看懂的内容：支付多少 token、收到什么、哪些授权改变、哪个合约被调用、失败损失多少；回答「这笔交易会让用户失去什么、得到什么、承担什么风险」 |
| Revocation | 中级 | 权限如何被及时收回——设计授权时必须同等重视撤销 | 用户主动撤销 + 系统自动收紧（session key 到期/额度用完/多次失败/检测异常地址/行为偏离任务/长时间未确认）；**原则：所有自动化权限都应该有用户看得见的关闭入口** |
| Human Check | 初级 | 在关键风险点让用户看懂并决定，而不是确认每一步 | 分层：低风险可逆→自动；中等→模拟后确认；高风险不可逆→必须展示影响后确认；超出 policy→直接拒绝；好的 Human Check 让用户看懂「批准的是什么、风险在哪里、为什么现在需要你确认」 |

**3. 在 AI × Web3 里的位置（20 分钟）**

稳定的链路：
```
用户给出目标和约束
→ Agent 读取上下文并生成计划
→ 系统把计划转成受限交易或操作
→ Guard 和 simulation 检查风险
→ 用户确认关键动作
→ Smart Account 或 Safe 执行
→ 日志记录 Agent 做过什么
```

最容易出问题的是**第 3 步到第 6 步**——很多项目把「模型生成计划」做得很顺，但没有认真处理交易模拟、权限校验、撤销入口和审计记录，结果 Demo 看起来自动化，真实使用时没有足够安全边界。

### 我的笔记（学完后填写）

**Handbook 说「Session Key 不是小号私钥，而是一组受限能力」——这两种理解在实际设计上会有什么不同？**
小号私钥是指另一个钱包私钥的完全开放功能和主要账户的session key，两者一个是真的的条件、权限可控制，一个是完全开放的小号私钥

> **补充（Claude）：** 方向正确。把这两种理解落到具体实现上差异更清楚：**把 Session Key 当成「小号私钥」**——实现时就是再创建一个普通 EOA 钱包，往里存少量余额，让 Agent 持有它的私钥。问题是：这个私钥能做任何普通钱包能做的事——转账给任意地址、approve 任意合约——只是被余额限制，不是被能力限制。私钥一旦泄露，攻击者也能任意操作，而且「用完再充值」的流程会让限制形同虚设。**把 Session Key 当成「一组受限能力」**——实现时是在 Smart Account 上注册一套链上规则：只能调用 A 合约的 swap 方法、单笔不超过 10 USDC、总额度 50 USDC、三天后自动失效。这些限制是**协议层的硬约束**，不依赖余额多少，私钥泄露了也只能在这些规则内操作，超出就会被 on-chain 拒绝。区别核心：前者依赖经济约束（余额少），后者依赖能力约束（规则硬编码在链上）。

**Guard 和 Policy 都是在拦截越界动作，它们的区别是什么？为什么两者都需要？**
policy是执行的细节控制，guard是执行的检查，区别就是policy限定了执行交易的细节部分，而guard主要是设计安全层面的检查，确保资金不会转错地址以及遭到mev攻击等，两者都需要是工程上安全、冗余层面不可或缺的设计理念

> **补充（Claude）：** 大方向正确，但 Guard 和 MEV 保护关系不大——Guard 是在执行前检查交易是否符合 Policy，不是防 MEV 的工具（MEV 是 mempool 层的问题，Guard 在应用层）。更准确的分工是：**Policy = 规则的声明层**（定义什么是允许的，比如「每天最多 10 USDC、只能调用白名单合约」）；**Guard = 规则的执行层**（在每笔交易真正出去前，用代码检查这笔具体交易是否符合 Policy）。类比：Policy 是公司的财务报销规则，Guard 是财务审批人在你提交报销单时逐项核对的过程。**为什么两者都需要**：Policy 单独存在时只是一段规则文字，模型可能解读错，也可能被 Prompt Injection 绕过；Guard 是确定性代码，不管模型说什么，只要交易参数不符合规则就拒绝。Policy 定义边界，Guard 在运行时强制边界——两者缺一个都不完整。

**还不清楚的问题：**
如果需要做一个agentic wallet，最起码、最基本需要实现的架构和功能是哪些？（最小限度下）

> **补充（Claude）：** Handbook 最小实践给了场景：用户允许 Agent 每天最多花 5 USDC 支付白名单 API。按这个场景，最小 Agentic Wallet 需要四件事串起来：① **Smart Account**（能承载规则的账户，比如基于 ERC-4337 的合约账户）——没有这个，后面的规则没地方写；② **Session Key + Policy**（定义 Agent 的能力边界：时间、额度、白名单地址/合约、方法限制）——这是「有限授权」；③ **Simulation + Guard**（每笔交易发出前预演结果、检查是否越界）——这是「安全执行」；④ **Revocation + Log**（用户能看到并关掉权限；每笔操作有可查记录）——这是「随时撤销 + 可追踪」。最小 demo 可以不用 Safe（多签是团队场景），不用复杂的 Human Check UI，但以上四点不能省。能展示「正常执行」「超额被拒」「地址不在白名单被拦」「用户撤销后失效」四个对比，就是一个完整的最小 Agentic Wallet。
